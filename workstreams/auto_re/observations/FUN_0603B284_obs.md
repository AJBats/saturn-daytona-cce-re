## Execution Path Analysis

### Section 1: Proximity Check (lines 1559-1607)

1. Calls function via `jsr @r0` (R0 loaded by caller, likely FUN_06047E0C = atan2)
2. Negates result → r10 (angle between the two cars)
3. Reads field[0x0E] from both R14 and R13 (heading values)
4. Subtracts r10 from each heading, wraps via extu.w
5. Compares against 0x8000 threshold (half rotation)
6. Uses results as indices into table at sym_002DD670 (24-byte entries)
7. Reads proximity value from table, subtracts R12, adds table result for other car
8. If result <= 0 (cmp/pl r4 at line 1604): early exit via `bra .L_0603B484`

### Section 2: Angle Range Check (lines 1615-1646)

If proximity check passes:
1. Reads field[0x0E] from both structs
2. Subtracts computed angles, checks if within 0x4000 range
3. If angle difference is large (between 0x4000 and 3×0x4000):
   - Compares field[36] values against threshold 0x0003B425
   - If exceeded: writes flag 0x80000000 to output struct A, offset +24

### Section 3: Secondary Angle Check (lines 1658-1686)

1. Compares field[36] difference against 0x0003B425
2. If exceeded: writes flag 0x80000000 to output struct A, offset +24
3. Checks field[0x0E] of R14 against angle threshold
4. If out of range: skip to collision response (Section 5)

### Section 4: Distance-Weighted Values (lines 1698-1750)

If all angle checks pass:
1. Writes heading values to output struct A offsets +16 and +4
2. Calls trig function (pool .L_pool_0603B504)
3. Computes field[36] delta between structs, scales by trig result
4. Writes weighted delta to output struct A offset +20
5. Scales field[36] of R14 by 0x0001CCCC (pool .L_pool_0603B508)
6. Writes to output struct B offset +20

### Section 5: Collision Response Vectors (lines 1751-1827)

Common path after all checks:
1. Computes collision direction using positions from R13 (struct B)
2. Calls trig functions (FUN_06047D3C and FUN_06047D20 via pools)
3. Scales results by the proximity value
4. Writes sin/cos components to output struct A offsets +8 and +12
5. Repeats with positions from R14 (struct A)
6. Writes sin/cos components to output struct B offsets +8 and +12
7. Calls function at pool .L_pool_0603B518 for BOTH output structs

## Other Observations

- FUN_0603B284 is NOT part of the per-car physics pipeline (FUN_0603DF28.s).
  It runs in a separate collision detection pass dispatched by FUN_0603976C
  in FUN_06038DD8.s. The call stack shows a completely different path from
  the physics pipeline observations.

- GBR at entry is 0x06057800 — NOT a chain entry pointer. The collision system
  uses its own context structure separate from the per-car GBR dispatch.

- The function processes PAIRS of cars (R14 and R13 are two different chain
  entries). The output is written to TWO global structs (sym_06052850 and
  sym_0605286C), each 28 bytes, containing collision response data for the
  pair.

- The proximity check (Section 1) uses a table at sym_002DD670 with the
  angle difference as an index. This table maps angle-to-proximity, allowing
  cars that are approaching head-on to have a larger collision envelope than
  cars moving in parallel.

- FUN_0603B4A4 (line 1854) has nearly identical code to B284 but is entered
  from FUN_0603B498 (line 1844) which computes coordinate differences from
  R8/R9 and R6/R7 before calling the atan2. B4A4 uses the same angle
  comparison and collision response logic.

- The output struct at offset +24 is a flag field. When the angle check at
  Section 2 detects cars approaching from certain directions AND field[36]
  exceeds 0x0003B425, the flag 0x80000000 is OR'd into offset +24.

- R12 decreases over successive frames (0x4559C → 0x38C99 → 0x335DA),
  consistent with two cars converging. Once R12 reaches a threshold where
  the proximity check fails (table value - R12 + table value <= 0), B284
  would exit early.

- The function is inactive during idle (B1=0 in CDL) because cars at the
  start grid are too far apart and moving slowly. With throttle, cars begin
  converging around frame 2334 (~7.8 seconds at 30fps).

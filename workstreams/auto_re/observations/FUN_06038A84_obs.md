## Behavior

FUN_06038A84 is the **track segment heading computer**. Per frame, for each car:

1. Reads car's segment index from `car[R0]` (R0=0xFFFFFFBF passed by caller)
2. Copies car X/Y/Z into a track history table at 0x060527DC (stride 20 bytes)
3. Calls FUN_0602AAA8 (segment lookup) — determines which track segment
   the car is in, writes result to car struct +0x1A4
4. Reads the segment pointer from car struct +0x154
5. From that segment struct, reads direction vector at +0x24 (X) and +0x28 (Z)
6. Calls FUN_06047E0C (atan2) with (-Z, X) to compute track heading
7. Writes result to car struct +0x194 (track heading angle)

## Key Discovery: CS0_BLK.BIN

The segment pointer at car struct +0x154 points into CS0_BLK.BIN data:

| Property | Value |
|----------|-------|
| Disc file | DAYTONA/CS0_BLK.BIN |
| Load address | 0x060ED100 (HWR) |
| Size | 42,676 bytes (0xA6B4) |
| RAM match | 99.9% (6/10,669 dwords differ — endian fixups) |
| Segment stride | Variable (pointer-based, not fixed stride) |
| Key offsets per segment | +0x02 (type), +0x24 (dir X), +0x28 (dir Z) |

This is the **track path/waypoint data** — the "second gear" complementing
CS0_COL.BIN (collision polygons). COL defines surface geometry for collision.
BLK defines track direction for AI path following and heading computation.

## Attract Mode Path Following

During attract mode:
- Steering input (+0x78) is zero — unused
- FUN_06038A84 computes heading from BLK segment direction vectors
- The heading is written to +0x194 (track heading angle)
- Something downstream reads +0x194 and updates the physics heading +0x38

## Transplant Implications

For DUSA driving model transplant, BOTH disc files must be replaced:
- CS0_COL.BIN (112K) — collision polygons, grid index
- CS0_BLK.BIN (42K) — track segments, AI path, heading data

Combined: ~154K per course. The BLK data is critical for AI cars and
potentially for the rolling start path as well.

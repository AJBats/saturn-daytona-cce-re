## Execution Path Analysis

FUN_0603F166 has two main paths:

### Path A: Early Return (ALWAYS taken in tested scenarios)

The function has two loops (r7=2 iterations each). In each loop iteration:
1. Calls FUN_06036AA8 via JSR (pool at .L_pool_0603F48C)
2. Calls FUN_0603F10E with r0 = return value of FUN_06036AA8

FUN_0603F10E (lines 63-67):
```
tst r0, r0      ; T = (r0 == 0)
bt FUN_0603F116  ; if r0 == 0, go to F116
rts              ; if r0 != 0, return (T=0)
nop
```

FUN_0603F116 (lines 72-88):
- Modifies GBR+76 (adds ±0x00008000 depending on sign)
- Calls FUN_0603ECF0 with r5 = GBR+12 (unsigned)
- Returns with `sett` (T=1)

In the first loop: `bf .L_0603F1DE` (line 181) — if T=0, take processing path.
Since T is always 1 (FUN_06036AA8 always returns 0), bf is never taken.
The function falls through to .L_0603F1D2 and returns immediately.

A breakpoint at .L_0603F1DE (0x0603F1DE, the processing path) was never hit
in 6000 frames with throttle held.

### Path B: Processing (NEVER observed)

If FUN_06036AA8 returns nonzero, F10E returns T=0, and the function enters
the processing path. This path:
- Accumulates a bit pattern in r8 via `rotcl r8` (lines 194, 257)
- Writes r8 to GBR+140 (line 277)
- If r8==0: computes values and writes to R14+4, R14+12, R14+14, R14+16
- If r8!=0: walks backward through stack to find last valid entry
- Scans results for value==2, writes 0 or 1 to GBR+19 (line 374)
- If GBR+140!=0: enters a complex post-processing path that reads GBR+192,
  GBR+72, GBR+144, modifies GBR+76, and calls FUN_0603F0B4, FUN_06044D80,
  FUN_0604507E, FUN_0603EC40, FUN_0603ECF0

A watchpoint on GBR+140 (0x060FFA8C) for entry 38 showed zero hits from
frame 2015 through frame 3000.

## Other Observations

- FUN_0603F166 is a large function (0x0603F166-0x0603F458, 754 bytes) but
  in practice always takes the early return after 1 loop iteration. The
  vast majority of the function body (Path B) was never executed in any
  tested scenario.

- The function allocates a large stack frame (-148 bytes, pool 0xFFFFFF6C).
  The local stack is organized as 4 groups of 5 longwords (0x14 bytes each),
  addressed via r12 (starts at stack+0x44).

- The two loops (r7=2 each, 4 total iterations if fully executed) appear to
  process lookahead points. Each iteration reads a 16-bit value from a table
  pointed to by r13 (pool .L_pool_0603F480 = 0x0603F33C, which is inside
  F166 itself at line 373-378 area), adds the current GBR+12 value, and
  calls FUN_06047D3C and FUN_06047D20 as coordinate transform functions.

- The dmuls.l + xtrct pattern (lines 153-157, 164-167, etc.) is a 32x32→32
  multiply taking the middle 32 bits: (a * b) >> 16. The multipliers
  0x00020581 and 0x0001FA5E are close to 2.0 and ~2.0 in 16.16 fixed point
  (0x00020000 = 2.0, so these are slightly above and below 2.0).

- FUN_06036AA8 (the gate function) consistently returns 0 for entry 38,
  meaning the "check" always passes and the function returns early. The
  check returns 0 for all observed calls (entry 38 at tier 4).

- The Verifier's 1/3 pass result (call_count claim only) matches: F166 is
  called once per frame when a tier 4-5 car exists. The processing path
  writes would need specific trigger conditions not achievable in standard
  test scenarios.

- FUN_0603F134/F146 are callee-save wrappers that call F166 with GBR set
  from R4 (an arbitrary chain entry pointer). They have no external callers
  in the race module's BSR/JSR references. They may be called via indirect
  dispatch or may be dead code.

- The PC trace consistently shows the same dispatch pattern across frames:
  15 hits at E7B0 (tier 0-1), 4 at E9E2 (tier 2), 1 at EAAA (tier 3),
  0 at E952 (tier 4-5) in most frames. The 15+4+1=20 dispatches correspond
  to approximately 20 active chain entries per frame.

## Other Observations

- R0 = R4 = R14 = GBR = 0x0605224C at entry in ALL scenarios. The caller sets up
  all four registers to point to the player struct before calling.
- R13 is used as the callee address register throughout — loaded from pool, then `jsr @r13`.
- The function saves/restores r9-r14 + pr, using 7 stack slots.
- Two conditional branches check GBR+0x174 and GBR+0x16A to select between
  alternate sub-function calls.
- The caller at 0x06037F3E is in the main per-frame game loop.
- This function does NOT appear in the per-car AI loop (FUN_0603EE64).
- The player struct at 0x0605224C is active in BOTH time trial and race mode,
  with identical register context. It is completely separate from the 40-entry
  chain array at 0x060FD400.
- R5=0x060540F4 and R13=0x0605278C suggest additional data structures nearby
  the player struct (0x0605278C = player struct + 0x540 = possible extended data).
- Field +0x98 (in chain array context = tier) reads 0x00000038 in most scenarios
  but 0x000000B8 with brake. In the player struct this offset has a different meaning
  than in the chain array (where it holds tier identifiers like 0x01010101).
- sample_fields.py crashes at ~frame 100 with PermissionError on Windows (IPC file
  lock race in MednafenBot.send). Acceleration curve was sampled via MCP tools instead.
